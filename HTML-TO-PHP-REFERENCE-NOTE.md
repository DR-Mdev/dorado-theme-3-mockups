# HTML → PHP(WordPress テーマ化)移行リファレンスノート

作成日: 2026-07-21
状態: **参考メモ。着手は全モックアップHTMLの最終版が固まってから。** このファイル自体は移行の実装ではなく、次に移行作業へ入るときに読む前提知識・全体像・進め方の提案。

---

## 背景

`use-case-single.html` / `insight-single.html` を中心に、「このHTMLのままではWordPressの編集画面(投稿編集・カスタムフィールド)で安全に入力・運用できるのか」を検証した際の結論をまとめたもの。個別ページのレビューで見えた論点を、サイト全体のテーマ化に一般化して記録する。

---

## 核心の結論

**HTMLをどれだけ整備しても、PHP化(WordPressテーマ化)の作業そのものはゼロにならない。** ただし区別すべき2種類の「事前準備の要否」がある:

- **リデザイン**(例:index.htmlの再設計)= 旧HTMLから読み取れない新しい意思決定(セクション削除・移動・新規コピー)を実装者に渡す作業。**ページ個別のspec(Markdown)が必要。**
- **PHP化(変換)**= 既に固まったデザインを、見た目を保ったままCMSの仕組みに載せ替える機械的マッピング作業。**ページ個別specではなく、サイト共通の「変換規約」1本で駆動できる。**

「specがないと必ずおかしくなる」のではなく、正確には**「規約(HTMLに埋め込む注釈+共通コンベンション文書)がないとおかしくなる」**。モックアップには既に `<!-- WP: the_content() -->` `<!-- WP: the_post_thumbnail() -->` `<!-- WP LOOP START -->` 等の注釈が入っており、作者は最初から変換を意識して書いている。ギャップは想定より小さい。

---

## ① 今(HTML側で)できること — 変換を速く・壊れにくくする

以下はすべて「あいまいさの削減」と「重複の排除」であり、**やっても②の作業量そのものは減らない**。ただし変換時の判断コストと崩れるリスクを下げる。

1. **WP注釈コメントの標準化**
   一部のファイルにしかない `<!-- WP: … -->` を全ファイルで統一し、動的な箇所すべてに「どのWP関数/フィールドか」を明記する(`the_title()` / `the_content()` / ACFフィールド名 / `WP_Query` の引数など)。**最も効果が大きい対策。** 静的HTMLは全部が等しく「ベタ書き」に見えるため、どこが固定テンプレでどこが編集領域かを注釈が示すだけで、変換者(人でもAIでも)が推測しなくて済む。

2. **共通パーツ(partial)の同一性を保証**
   header / footer / SP-nav / `<head>` のトークン定義 / contact CTA / カード(`.ins-card`)は、現状すべてのファイルに複製されている。テーマでは `header.php` `footer.php` `template-parts/card.php` 等に一本化されるので、**今のうちに各コピーがバイト単位で同一かを確認・統一**しておくと、そのまま1枚として抜き出せる。少しでも各ページで食い違っていると、抽出時に「どちらが正か」の判断が発生する。

3. **クラス名と構造の固定 = デザイン契約**
   CSSが要素+クラスベースで各ページに共有されているため、クラス名を全ページで揃えておけば1本のスタイルシートで全テンプレートを賄える。これが「標準ブロックを打ってもデザインが崩れない」ことの前提であり、**変えないこと自体が準備**になる。

4. **コンテンツモデルのヒント付け(軽微)**
   use-caseの「At a glance」行や、サイドバーTOCなど「これはカスタムフィールド駆動/自動生成である」とわかる `data-` 属性やコメントを添えておく。

**この①の性質:** すべて「注釈とハウスキーピング」であり、CMSの中身(CPT・ACF・テンプレート)は一切作らない。やると変換フェーズが楽になるだけ。

---

## ② テーマ化で必ず必要になること — HTMLでは絶対に消せない

静的HTMLには対応物が存在しない「CMSの配管」であり、**HTMLをどれだけきれいにしても必ず発生する**作業。

1. **テンプレート階層ファイル**
   `header.php` / `footer.php` / `front-page.php`(index)/ `page.php` + 固定ページテンプレート(about・services・how-we-operate・partners・contact)/ `archive-*.php`・`taxonomy-*.php`(insights・use-cases 一覧+チップ絞り込み)/ `single-*.php`(insight・use-case の2種)。

2. **カスタム投稿タイプ + タクソノミー登録**
   `insight` / `use_case` 投稿タイプ、`insight_category` / `usecase_stage` タクソノミー。チップによる絞り込みとループ表示はこれが存在して初めて動く。

3. **カスタムフィールド(ACF等)**
   At a glance(Client / Objective / Scope / Duration / Deliverables)、dek、read time、sector、duration、stage番号など。

4. **本文オーサリング層**
   `pull-quote` / `note-bar` を編集画面から挿せるようにする(ブロックパターン、またはACFフレキシブルコンテンツ)。

5. **動的クエリ**
   関連記事表示、一覧ループ、ページネーション、カテゴリーチップのリンク生成(`WP_Query`、`tax_query`)。

6. **TOC自動生成**
   本文中のh2見出しから `id` とサイドバーTOCを自動生成する小さなJSまたはPHPフィルタ。手書きTOCのままだと、編集者がh2を追加・削除・改名した際にアンカー切れが起きる。

7. **メニュー登録・アセット読み込み**
   ナビのWPメニュー化、フォント/favicon/共通CSS・JSの `wp_enqueue_*`。

**この②の性質:** すべてバックエンド/テンプレートの実装。①をどれだけやっても総量は減らない。

## サイト全体像と移行の進め方(提案)

サイトは約13ファイル構成。**全ファイルの精査は必要だが、リデザインではなく「構造監査」として一度だけ**行う想定。

### 推奨する順序

1. **監査パス(精査)**
   全ファイルで以下を確認する:
   - (a) 共通パーツ(header/footer/SP-nav/カード等)が全ファイルで同一か
   - (b) 動的領域(投稿データ・ループ・フィールド)の棚卸し
   - (c) 必要なCPT / タクソノミー / カスタムフィールドの洗い出し

   成果物は**「サイト共通変換規約」1本 + テンプレート別コンテンツモデル表**。これが揃えば、以降はページ個別のspecを作らずに変換を進められる。

   > 監査時に確定させるべき論点の例: カードCSSはページごとに微妙にサイズが違う(例: insights一覧の `.ins-body` は `padding:20px`・`h3{font-size:1.08rem}`だが、single側の関連記事カードは `padding:18px`・`h3{font-size:1rem}`)。これは意図的なサイズ差と見られるため、共通カードコンポーネントに「文脈修飾クラス」(archive用/related用など)を持たせる設計になる。

2. **デザイン凍結**
   見た目が全ページで固まった時点で、各ファイルに重複しているインラインCSSを1本の `style.css` に単一ソース化する(監査で全ファイルのCSSが同一であることを確認した上で)。

3. **テーマ骨組みの実装**
   header.php / footer.php / functions.php(`wp_enqueue_*`・CPT登録・タクソノミー登録・メニュー登録・ACFフィールドグループ・TOC自動生成・ブロックパターン登録)。

4. **1枚パイロット変換**
   最も静的なページ(`contact` または `about`)を先に `page.php` 化してパイプラインを検証 → 次に一覧系(archive: insights・use-cases)→ 次にsingle系(最も動的: insight-single・use-case-single)→ 最後に front-page(index)。

5. **QA**
   デザイン契約(共通CSS)が保証されているため、各ページはモックアップとの視覚差分チェックのみで済む想定。

この順番なら、「①のHTML整備」は監査パスの副産物として自然に片付き、「②の配管」はテーマ骨組みの段階で一括対応でき、個々のページ変換はパイロットで型が確定してから量産フェーズに入れる。
